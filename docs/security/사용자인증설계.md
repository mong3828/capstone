## 사용자 인증 설계

### 의도
- 3주차에 로그인/JWT를 구현하기 전에, **토큰에 무엇을 담고**, **만료/갱신을 어떻게 할지**를 먼저 설계한다.
- object-level auth는 결국 `subject(user_id, tenant_id, role)`을 안정적으로 얻는 문제이므로,
  JWT의 클레임(Claims)을 2주차에 확립할 예정이다.

---

## 1) Access Token(JWT) 기본
### 헤더 전달
- `Authorization: Bearer <access_token>`

### 만료
- **expires_in**: 1시간(3600s)

### 예상되는 필수 클레임
- `sub`: user_id
- `tid`: tenant_id
- `role`: `admin` | `user`
- `iat`: issued at(발급 시간)
- `exp`: 만료 시간
- (선택) `jti`: 토큰 ID(추후 블랙리스트/감사로그 연결용)

---

## 2) Refresh Token (선택, 3~4주차에 결정 예정)

### 옵션 A: Refresh Token 미사용
- access token 만료 시 재로그인
- 구현이 단순하고, 보안/저장 이슈 최소

### 옵션 B: Refresh Token 사용
- refresh token은 DB에 해시로 저장 + 로테이션
- access token 만료 시 `/auth/refresh`로 재발급

---

## 3) 오류 코드 정책 (3주차 기중)
- **401**
  - 토큰 없음
  - 토큰 만료
  - 서명 검증 실패
- **403**
  - 토큰은 유효하지만, object-level auth 실패(남의 Job에 접근 등)

---

## 4) 로그/보안 주의사항
- JWT 원문을 로그에 남기지 않는다(필요 시 `sub/tid/jti`만 남김)
- pre-signed URL과 마찬가지로 민감정보는 마스킹 규칙을 적용한다
