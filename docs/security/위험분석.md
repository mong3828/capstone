- 표기 규칙
  - **MUST**: MVP에서 반드시 통과해야 함(4주차 수직슬라이스 기준)
  - **SHOULD**: 6~7주차까지 목표(가능하면 앞당김)
- 현재 AC-06, AC-07은 해결 완료

### AC-01. IDOR 방지(object-level auth)

- **MUST**: 인증된 사용자 A가 사용자 B의 `job_id`를 알고 있어도, B의 Job을 조회/다운로드할 수 없어야 한다.
  - **검증 방법**
    - 사용자 A로 로그인 후 Job 생성(또는 JobID 확보)
    - 사용자 B로 로그인한 토큰으로 `GET /jobs/{job_id}` 호출
    - **기대 결과**: 403(Forbidden)
- **MUST**: “존재하지 않는 Job”과 “권한 없는 Job”을 구분해 정보 유출이 생기지 않도록, 외부 노출 정책을 정한다.
  - **권장 정책**: 권한 없음은 403, 미존재는 404(단, ID 열거 위험이 크면 404로 통일 가능)

### AC-02. CSV Injection 방어

- **MUST**: 업로드/처리 과정에서 수식 인젝션 패턴이 탐지되면 차단하거나 안전화(이스케이프)해야 한다.
  - **검증 방법**
    - `=HYPERLINK("http://evil")` 또는 `@SUM(1,1)` 등이 포함된 CSV 샘플 준비
    - 업로드 → 처리 → 결과 다운로드
    - **기대 결과**: (정책 선택에 따라) 업로드가 4xx로 거절되거나, 결과 CSV에서 위험 셀이 무해화됨

### AC-03. 시스템 자원고갈 방지(업로드 제한)

- **MUST**: 파일 크기 제한이 존재하고, 초과 시 업로드가 거절되어야 한다.
- **MUST**: 행/열 또는 처리 타임아웃 정책이 문서화되어야 한다.
  - **검증 방법**
    - 제한값을 넘는 파일/요청을 시도
    - **기대 결과**: 413(또는 400) + 정책에 맞는 에러 코드/메시지

### AC-04. Pre-signed URL 안전성

- **MUST**: pre-signed URL에는 만료 시간이 적용되어야 한다.
- **MUST**: 서버 로그/감사로그에 pre-signed URL의 서명 파라미터 또는 토큰이 그대로 남지 않아야 한다.
  - **검증 방법**
    - URL 발급 후 로그 확인(로컬 콘솔/파일)
    - **기대 결과**: 토큰/서명 파라미터가 마스킹되거나 기록되지 않음

### AC-05. 키 유출 방지

- **MUST**: 레포에 비밀정보가 포함된 파일이 커밋되면 안 된다.
  - 예: `.env`, `credentials.json`, `*.pem`, `id_rsa`, AWS access key가 들어간 파일
- **MUST**: CI 로그에 비밀정보가 출력되면 안 된다.
  - **검증 방법**
    - 레포 전체 검색(로컬) + PR 리뷰 체크리스트
    - CI 실행 로그에서 비밀 값이 마스킹되는지 확인

- **MUST**: 모든 API 응답에 `x-request-id`가 포함되어야 한다.
- **MUST**: 최소한 “누가(사용자), 무엇을(액션), 언제, 어떤 리소스(Job/File)”를 남길 수 있는 구조가 있어야 한다.
  - **검증 방법**
    - 주요 API 호출 후 로그/저장소에 기록 확인
    - **기대 결과**: request_id가 연결되어 추적 가능

### AC-07. 내부 정보를 노출하지 않는 오류 메시지
<img width="907" height="416" alt="image" src="https://github.com/user-attachments/assets/444ca3b9-43ae-4b1a-861b-1976b1cc9a20" />

- **MUST**: 500/예외 상황에서 스택트레이스/파일 경로/비밀값이 응답에 포함되면 안 된다.
  - **검증 방법**
    - 의도적으로 예외 유발(예: 잘못된 입력)
    - **기대 결과**: 표준화된 에러 payload + request_id만 반환
