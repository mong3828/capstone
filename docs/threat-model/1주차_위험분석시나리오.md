## 1주차 위험분석 시나리오
작성자: 2376292 최승

### 1) 범위/가정(Week 1)
- **목표 시스템**: 사용자 인증을 거쳐 CSV를 업로드하고, 서버가 워터마크 처리를 수행한 뒤 결과 CSV를 다운로드하는 서비스.
- **Week 1 범위**: 위협 시나리오/자산/신뢰 경계(Trust Boundary) 정의 및 보안 수용 기준(AC) 초안 고정.
- **가정**
  - 현재는 개발 초기 단계이므로 로컬 실행 중심이지만, 3~4주차부터 클라우드(S3/DB) 연동을 목표로 한다.
  - 클라우드 연동 시 업로드는 **pre-signed URL** 기반(서버 경유 최소화)을 기본으로 한다.

### 2) 보호해야 할 자산(Assets)
- **A1. 사용자 계정/인증 정보**: 비밀번호 해시, JWT/리프레시 토큰(도입 시)
- **A2. 원본 CSV 데이터**: 민감 데이터 포함 가능(개인정보/기업 데이터)
- **A3. 결과 CSV 데이터**: 워터마크/검출 관련 결과
- **A4. Job 메타데이터**: JobID, 상태, 소유자(user_id/tenant_id), object_key(원본/결과)
- **A5. 감사로그(AuditLog)**: 누가 무엇을 했는지(보안/운영 증적)
- **A6. 비밀정보(Secrets)**: AWS 키, JWT 서명키, DB 접속 정보, CI 토큰

### 3) 신뢰 경계(Trust Boundaries)
- **TB1. 클라이언트 ↔ API 서버**: 인터넷/외부 네트워크 경계(인증/인가/입력검증)
- **TB2. API 서버 ↔ 스토리지(S3)**: pre-signed URL 발급/정책/만료가 핵심
- **TB3. API 서버 ↔ DB**: 권한 모델의 단일 근거(소유자/tenant)
- **TB4. 처리 실행(워커/엔진) ↔ 파일 시스템/스토리지**: 악성 입력/자원 고갈/임시파일 관리
- **TB5. CI/CD ↔ 레포/비밀정보**: 키 유출, 공급망 위험

### 4) 핵심 위협 시나리오
공격자의 목표 → 진입점 → 공격 방법 → 영향 → 방어/검증(AC 연결) 형식으로 작성함.

#### T1. IDOR(수평 권한 상승): JobID만 알아내 남의 결과 다운로드
- **목표**: 다른 사용자의 결과/원본 데이터 탈취
- **진입점**: `GET /jobs/{job_id}`, `POST /jobs/{job_id}/download-url`
- **방법**: job_id 추측/유출 후 직접 호출
- **영향**: A2/A3/A4 유출(심각)
- **방어**: object-level auth(소유자/tenant) 필수 + 403 일관 처리
- **검증**: 다른 사용자 토큰으로 접근 시 항상 403 (AC-01)

#### T2. CSV Injection(수식 인젝션): 결과 CSV 열람 시 클라이언트에서 명령 실행 유도
- **목표**: 다운로드 후 엑셀/스프레드시트에서 수식 실행 유도, 정보 유출/피싱
- **진입점**: 업로드 CSV 콘텐츠
- **방법**: `=HYPERLINK(...)`, `=cmd|...` 등 셀 선두에 수식/특수문자 삽입
- **영향**: 사용자 환경에서 추가 피해(중요)
- **방어**: 위험 패턴 탐지/이스케이프(선두 `= + - @` 등) + 정책 문서화
- **검증**: 악성 샘플 업로드 시 차단 또는 안전화 처리 (AC-02)

#### T3. 대용량 업로드/처리 DoS: CPU/메모리/시간 고갈로 서비스 마비
- **목표**: 서비스 다운/비용 유발
- **진입점**: 업로드(파일 크기/행/열), 처리 실행
- **방법**: 초대형 CSV, 압축폭탄(추후), 매우 긴 행/열, 반복 요청
- **영향**: 가용성 저하, 큐 적체, 비용 증가
- **방어**: 크기/행/열 제한, 타임아웃, rate limit(6주차 확대)
- **검증**: 제한 초과 시 413/429/400 등 정책대로 거절 (AC-03)

#### T4. Pre-signed URL 유출/재사용: 만료 전 제3자가 업로드/다운로드
- **목표**: 데이터 오염/탈취
- **진입점**: pre-signed URL(쿼리스트링)
- **방법**: 로그/브라우저 히스토리/스크린샷/공유로 URL 유출
- **영향**: A2/A3 오염 또는 유출
- **방어**: 짧은 만료, 최소 권한(prefix/객체 단위), URL을 로그에 남기지 않음(마스킹)
- **검증**: 만료 후 업/다운로드 실패 + URL/서명 파라미터가 로그에 남지 않음 (AC-04)

#### T5. 키 유출: 레포/로그/CI에서 AWS 키 또는 JWT 비밀키 노출
- **목표**: 인프라 장악, 데이터 대량 유출
- **진입점**: Git 커밋, CI 로그, 설정 파일
- **방법**: 실수로 `.env` 커밋, 로그에 키 출력, CI 환경변수 노출
- **영향**: A6 유출 → 전체 시스템 붕괴
- **방어**: 비밀정보 커밋 금지, 최소권한 IAM, 키 로테이션, 로그 마스킹
- **검증**: 레포에 `.env`/키 파일 없음, CI 로그에 비밀값 출력 없음 (AC-05)

#### T6. 로그 위변조/부재: 사고 발생 시 추적 불가(증적 부족)
- **목표**: 공격 흔적 은폐 또는 운영 장애 분석 불가
- **진입점**: 서버 로깅/감사로그 설계
- **방법**: 요청ID 부재, 사용자/액션 미기록, 로그가 민감정보로 오염되어 활용 불가
- **영향**: 대응 지연, 원인 규명 실패
- **방어**: 요청ID 전파, 최소 감사로그 스키마, 민감정보 마스킹
- **검증**: 주요 API 호출에 request_id가 항상 존재, 액션 로그가 남음 (AC-06)

#### T7. 입력 파싱 취약점: 악성 CSV로 파서/엔진 예외 유발 및 정보 노출
- **목표**: 서비스 중단, 내부 정보(스택트레이스) 노출
- **진입점**: 업로드/처리 단계
- **방법**: 깨진 인코딩, 비정상 구분자, 매우 긴 필드, 특수문자
- **영향**: 500 다발, 내부 경로/라이브러리 정보 노출
- **방어**: 에러 응답 표준화(스택트레이스 미노출), 파싱 예외 분류, Fail 상태 기록
- **검증**: 잘못된 CSV는 4xx 또는 Job Fail로 처리되고, 사용자 응답에 내부 정보가 없음 (AC-07)

### 5) 오픈 이슈(Week 2로 넘길 것)
- 멀티테넌시 모델(tenant_id) 확정 및 인덱싱/쿼리 패턴(“권한 체크가 빠르게 가능해야 함”)
- 토큰 만료/갱신 전략(리프레시 토큰 도입 여부)
- CSV Injection의 “차단 vs 이스케이프” 정책 선택
