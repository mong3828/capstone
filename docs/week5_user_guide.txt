================================================================================
B2MARK 5주차 — 사용자(운영자) 수행 작업 가이드
================================================================================
이 문서는 코드 저장소에 포함된 배포·운영 단계에서 **본인이 AWS 콘솔·GitHub 등에서
직접 수행**해야 하는 작업을 순서대로 정리한 것입니다. (개발팀이 자동화한 부분과
구분됩니다.)

--------------------------------------------------------------------------------
1. 사전 준비
--------------------------------------------------------------------------------
  • AWS 계정 및 결제(또는 학생/크레딧) 설정
  • 도메인을 쓸 경우 Route 53 또는 외부 DNS 접근 권한
  • GitHub 저장소에 푸시 권한 (CI/CD 시)

--------------------------------------------------------------------------------
2. IAM (최소 권한 원칙)
--------------------------------------------------------------------------------
  1) IAM 사용자 또는 역할을 만든다. (루트 계정으로 일상 작업 금지)
  2) 배포에 필요한 권한만 붙인다. 예:
     - ECR: 이미지 push/pull (AmazonEC2ContainerRegistryPowerUser 등, 팀 정책에 맞게 축소)
     - Lambda 또는 ECS: 해당 서비스 배포·업데이트
     - API Gateway: REST/HTTP API 생성·배포
     - Secrets Manager 또는 SSM Parameter Store: 시크릿 읽기 (런타임만)
  3) 액세스 키는 로컬에만 두지 말고, CI에서는 **GitHub Secrets**에만 저장한다.
     이름 예: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_REGION

--------------------------------------------------------------------------------
3. 컨테이너 레지스트리 (ECR)
--------------------------------------------------------------------------------
  1) ECR에 리포지토리 생성 (예: b2mark-api).
  2) 로그인:
       aws ecr get-login-password --region <리전> | docker login --username AWS --password-stdin <계정ID>.dkr.ecr.<리전>.amazonaws.com
  3) 로컬에서 이미지 빌드 후 태그·푸시:
       docker build -t b2mark-api .
       docker tag b2mark-api:latest <URI>/b2mark-api:latest
       docker push <URI>/b2mark-api:latest
  4) 이후 Lambda(컨테이너) 또는 ECS 태스크 정의에서 위 이미지를 참조한다.

--------------------------------------------------------------------------------
4. API 실행 방식 선택 (팀/예산에 따라 하나 선택)
--------------------------------------------------------------------------------
  A) AWS Lambda + API Gateway (HTTP API)
     - 컨테이너 이미지 또는 zip 배포. FastAPI는 Mangum 등 어댑터가 필요할 수 있음.
     - API Gateway에서 Lambda 연결, 스테이지 배포 후 **호출 URL** 확보.

  B) ECS Fargate + Application Load Balancer
     - 태스크 정의에 Docker 이미지, CPU/메모리, 포트(8000) 설정.
     - 타겟 그룹 헬스체크 경로: GET /health 권장.

  C) EC2 + docker run (간단 프로토타입)
     - 보안 그룹에서 인바운드 8000(또는 443)만 허용.
     - 환경 변수로 시크릿 주입 (아래 6항).

--------------------------------------------------------------------------------
5. API Gateway / 프론트엔드 연동
--------------------------------------------------------------------------------
  • 브라우저에서 호출할 **공개 Base URL**을 정한다 (예: https://api.example.com).
  • 프론트엔드(Vite/React 등)에는 빌드 시점 변수로 주입한다:
      VITE_API_BASE_URL=https://api.example.com
    (프로젝트의 실제 변수명은 프론트 코드의 import.meta.env / process.env 규칙에 맞출 것.)
  • CORS: API가 다른 오리진에서 호출되면 FastAPI에서 CORS 미들웨어 허용 목록에
    프론트 도메인을 추가해야 한다. (필요 시 개발자에게 요청.)

--------------------------------------------------------------------------------
6. 시크릿·환경 변수 (절대 저장소에 커밋 금지)
--------------------------------------------------------------------------------
  • B2MARK_SECRET_KEY (또는 프로젝트에서 사용하는 키 이름): 워터마크용 비밀키.
  • AWS 쪽: Secrets Manager에 저장 후 Lambda/ECS에서 참조하거나,
    SSM Parameter Store (SecureString) 사용.
  • 로컬 테스트용 `.env`는 `.gitignore`에 포함되어 있어야 하며, 팀원과는
    1Password 등 안전한 채널로만 공유.

--------------------------------------------------------------------------------
7. GitHub Actions (선택)
--------------------------------------------------------------------------------
  • 저장소 Settings → Secrets and variables → Actions 에 AWS 자격·리전·ECR URI 등을 등록.
  • 배포 워크플로는 팀이 별도로 작성하거나, `docker build` 검증만 사용할 수 있다
    (현재 `docker.yml`은 **이미지 빌드 검증**용).
  • 실제 `docker push` / ECS 배포 job은 IAM 권한과 OIDC 연동 등을 검토한 뒤 추가한다.

--------------------------------------------------------------------------------
8. 배포 후 확인 체크리스트
--------------------------------------------------------------------------------
  [ ] GET https://<API>/health → {"status":"ok"} (또는 동일 스키마)
  [ ] POST /watermark 샘플 CSV로 200 및 CSV 응답
  [ ] 수식으로 시작하는 셀(=1+1 등)이 있는 CSV는 400 거절 (5주차 보안)
  [ ] 대용량 업로드 시 413
  [ ] 프론트에서 실제 API URL로 요청 성공
